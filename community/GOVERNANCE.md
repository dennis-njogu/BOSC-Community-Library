## 🎯 Project Overview

**BOSC Community Library** is an open-source educational platform built collaboratively by the community. We believe in:

- 📖 **Open Access** — Education should be free for everyone
- 🤝 **Community-Driven** — Decisions made collectively
- 🌍 **Inclusivity** — Diverse voices and perspectives welcome
- 🔍 **Transparency** — All decisions and processes are documented
- ⚖️ **Fairness** — Equal opportunities for all contributors

## 👥 Team Structure

### Core Leadership

#### **Project Lead**
- Overall project direction and vision
- Final approval on major decisions
- External communications & partnerships
- Reports to stakeholders

**Responsibilities:**
- Set project roadmap
- Approve governance changes
- Represent project externally
- Ensure community health

#### **Maintainers** (2-3 people)
- Review pull requests
- Approve contributions
- Manage releases
- Maintain code quality standards

**Responsibilities:**
- Review all PRs within 7 days
- Approve/request changes on contributions
- Maintain documentation
- Guide contributors

#### **Community Coordinators** (1-2 people)
- Facilitate community discussions
- Organize meetings
- Document decisions
- Onboard new contributors

**Responsibilities:**
- Schedule & lead meetings
- Document meeting notes
- Welcome new members
- Facilitate discussions

### Contributors

**All Community Members** can contribute by:
- Adding educational resources
- Fixing bugs
- Improving documentation
- Translating content
- Suggesting improvements

## 🎓 Roles & Responsibilities

### Contributor Role
**Who?** Anyone who submits a contribution  
**Duration:** Ongoing  

**Responsibilities:**
- Follow Code of Conduct
- Follow Contributing Guidelines
- Submit clear PRs with descriptions
- Respond to feedback
- Help review others' contributions

**What You Can Do:**
- Submit PRs and issues
- Participate in discussions
- Help other contributors
- Suggest improvements

---

### Maintainer Role
**Who?** Experienced contributors (3+ months, 5+ merged PRs)  
**How?** Nominated and voted by existing maintainers

**Responsibilities:**
- Review PRs thoroughly within 7 days
- Approve or request changes
- Merge approved changes
- Maintain code/content quality
- Help guide contributors
- Attend maintainer meetings (monthly)

**What You Get:**
- Direct approval rights
- Input on project direction
- Recognition in project
- Elevated permissions on GitHub

**Requirements:**
- Active contributor for 3+ months
- At least 5 merged contributions
- Demonstrated understanding of project values
- Willingness to help others
- Commit to attending maintainer meetings

---

### Maintainer Lead Role
**Who?** Outstanding maintainer (6+ months, 10+ merged PRs)  
**How?** Elected by maintainer team annually

**Responsibilities:**
- Lead maintainer team
- Coordinate review process
- Final approval on contested PRs
- Lead maintainer meetings
- Report to Project Lead

**Responsibilities:** Same as Maintainer, plus:
- Schedule maintainer meetings
- Coordinate review assignments
- Mediate disputes
- Set quality standards

## 🤝 Decision-Making Process

### Major Decisions (Affects Project Direction)

**Examples:** Roadmap changes, governance updates, major technical decisions

**Process:**
1. **Proposal** — Anyone can propose in GitHub Issues or Discussions
2. **Discussion** — Community discusses (min 7 days)
3. **Voting** — Maintainers vote
4. **Implementation** — Decision documented and acted upon

**Voting Rules:**
- Maintainers + Project Lead vote
- Simple majority wins (>50%)
- Tie-breaker: Project Lead decides
- Votes documented in meeting notes

---

### Medium Decisions (Affects Workflows)

**Examples:** Process changes, new policies, contribution standards

**Process:**
1. **Proposal** — Suggested in Discussions
2. **Discussion** — 3-5 days community feedback
3. **Review** — Maintainer team reviews
4. **Decision** — Maintainers decide with input

---

### Minor Decisions (Day-to-Day)

**Examples:** PR approvals, content additions, bug fixes

**Process:**
1. **Proposal** — Submitted as PR/Issue
2. **Review** — Maintainer reviews
3. **Decision** — Maintainer approves/requests changes
4. **Implementation** — Changes made and merged

## 🔍 Review Process

### Pull Request Review

All contributions go through review process:

#### Timeline
- **Submitted** → PR created on GitHub
- **24 hours** → Assigned to reviewer
- **7 days** → Target review completion
- **3 days** → Respond to feedback
- **Approved** → Merge to main branch

#### Review Criteria
1. **Compliance** — Follows Code of Conduct & guidelines
2. **Quality** — Content/code quality standards met
3. **Documentation** — Changes documented clearly
4. **Testing** — Changes tested (if applicable)
5. **Alignment** — Aligns with project goals

#### Reviewer Responsibilities
- Provide constructive feedback
- Suggest improvements
- Approve or request changes
- Be respectful and helpful
- Complete review within 7 days

#### Contributor Responsibilities
- Respond to feedback within 3 days
- Make requested changes
- Ask for clarification if needed
- Show willingness to improve

#### Approval Path

```
Contributor submits PR
        ↓
Assigned to Maintainer
        ↓
Review (suggest changes or approve)
        ↓
Contributor responds
        ↓
If approved: Merge to main
If rejected: Discuss & revise
```

## ⚖️ Conflict Resolution

### Step 1: Direct Communication
- Discuss issue respectfully
- Try to find mutual understanding
- Document the discussion

### Step 2: Escalate to Maintainer
- Bring issue to maintainer team
- Present both perspectives
- Maintainer suggests resolution

### Step 3: Team Discussion
- Discuss at team meeting
- All parties present their view
- Team votes if needed

### Step 4: Project Lead Decision
- If unresolved, Project Lead decides
- Decision is final
- Documented in meeting notes

### Key Principles:
✅ **Respectful** — Even in disagreement  
✅ **Documented** — All steps recorded  
✅ **Transparent** — Process is open  
✅ **Fair** — Heard fairly  
✅ **Final** — Decision is made & respected

## 📅 Meetings & Communication

### Team Meetings
- **Frequency:** Monthly
- **Duration:** 1 hour
- **Participants:** Project Lead, Maintainers, Community
- **Purpose:** Discuss roadmap, decisions, community health

### Maintainer Meetings
- **Frequency:** As needed (min monthly)
- **Participants:** Maintainers + Project Lead
- **Purpose:** Coordinate reviews, discuss standards, plan

### Open Community Discussions
- **Format:** GitHub Discussions & Issues
- **Frequency:** Ongoing
- **Purpose:** Ideas, questions, suggestions

## 📝 Amendments to Governance

### Propose Changes To Governance:
1. Open GitHub issue with proposal
2. Discuss for 7+ days
3. Maintainers vote
4. If approved, update this document
5. Announce change to community
